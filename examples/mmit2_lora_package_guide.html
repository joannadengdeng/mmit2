mmit2 整包讲义（LoRA 第一版）

从整包 mental model 开始，围绕 LoRA 这一条主链，把 config → data → preprocess → train → eval → report 串成一份可以打印、可以回看、也能拿来继续写第二版的讲义。
读法建议：这份讲义不是“从一个函数的第 1 行读到最后 1 行”，而是先建立整包的脑内地图，再把真实样例输出挂回去。真正要先记住的是骨架：一条 CanonicalSample 如何沿着 LoRA 路线变成训练张量、梯度更新和评测结果。

这份讲义要解决什么

如果直接打开 src/mmit2/training/trainer.py 或 src/mmit2/training/preprocessors/chat_template.py，我们很容易把注意力卡在局部细节里，比如 prompt_mask、target_modules、device_map="auto"、apply_chat_template()。局部当然重要，但在整包层面，真正先要建立的是：这个库把什么样的输入，沿着什么样的路径，变成什么样的训练与评测行为。

一句话高层目标

mmit2 可以先看成一个“把多模态问答样本送进 Hugging Face VLM，再用某种 fine-tuning 方法训练，并把结果重新拿回来做本地评测”的流水线。我们这版只盯其中一条：LoRA 路线。

这份第一版为什么只讲 LoRA：LoRA 是最适合拿来立整包 mental model 的入口。它保留了完整训练主链，但方法本身又足够集中，不会一上来把注意力分散到量化、稀疏、干预或特殊 masking 上。

1. CanonicalSample 是什么

整包 mental model 里，第一个真正需要站稳的概念不是 tokenizer，也不是 optimizer，而是 CanonicalSample。因为它是“这个库内部约定的统一样本边界”。

src/mmit2/data/types.py（核心结构）
Turn:
    role: "human" | "assistant"
    content: str

CanonicalSample:
    id: str
    image_path: str
    turns: list[Turn]
    metadata: dict
为什么这个结构这么关键

因为从这一层开始，后面的代码不再关心 “VQAv2 原始列名是不是叫 question / answers / image”。后面只关心：我手里有没有一条统一样本，它有问题轮次、回答轮次、图片位置，以及附加 metadata。

字段在 LoRA 路线里的用途
id调试、日志、报错定位、评测输出
image_path用于训练预处理和推理时取图；HF 内存图像也会通过 metadata 带过去
turns构造成 Hugging Face chat template 的 messages
metadata放原始答案、PIL image 等不想丢掉但也不想提升为主字段的信息
Mental model：CanonicalSample 就是整包内部的“标准语言”。适配器负责把各个数据集翻译成它；预处理、训练、评测都依赖它。

2. Hugging Face processor / model 需要补哪些基础

这份代码大量依赖 Hugging Face 的多模态接口。这里先补齐几个最容易卡住的概念。

AutoProcessor

它不是单纯 tokenizer。对多模态模型来说，它通常同时负责：

  • chat template 渲染
  • 文本 tokenization
  • 图片预处理
  • 把文本张量和图像张量打包到一起
AutoModelForImageTextToText

这是 Hugging Face 上的一类“图文输入，文本输出”的模型接口。对我们来说，它最重要的行为是：

  • forward(..., labels=...) 可以直接返回 language modeling loss
  • generate(...) 可以直接做推理生成
apply_chat_template() 在这条链里扮演什么角色

它接收的是一种结构化的 messages，输出的是“该模型家族习惯的文本格式”。对于 Qwen2.5-VL，这个格式包含类似 <|im_start|>、<|vision_start|>、<|image_pad|> 这样的特殊 token。

一个很重要的细节：messages 并不是最终训练文本。它是“模型模板渲染前”的结构；full_text / prompt_text 才是渲染后的文本。

3. PyTorch 训练骨架需要补哪些基础

后面读 trainer.py 时，如果这些词没有先站稳，就很容易觉得代码在“神秘地自动完成训练”。

概念这份代码里它是什么意思
input_ids文本 token 的整数序列；在多模态模型里，图像也会对应到一串视觉占位 token
attention_mask哪些位置是真实 token，哪些只是 padding
labels监督目标；被设成 -100 的位置不会参与 cross-entropy
forward模型前向计算，得到 logits，很多 HF 模型还能顺手返回 loss
loss.backward()沿计算图回传梯度，写到可训练参数的 .grad
optimizer.step()用当前梯度更新参数
gradient_accumulation_steps每积累多次小 batch，再做一次真正参数更新
single sample tensors ↓ DataLoader + collate ↓ batch on GPU ↓ model(**forward_batch) ↓ loss ↓ loss.backward() ↓ if enough micro-batches: clip_grad_norm_ optimizer.step() scheduler.step() optimizer.zero_grad()
后面看到 labels[:prompt_len] = -100 时不要慌：它的意思不是“前面的 token 被删掉了”，而是“这些 token 仍然作为上下文输入模型，但不拿来算监督损失”。

4. LoRA 究竟改了什么

LoRA 不是“只更新模型里的一些现有大矩阵”，而是“在指定线性层旁边挂上一个很小的可训练低秩更新”。这件事对我们理解 lora.py、训练参数比例，以及为什么训练后还要 merge / unload，都很关键。

LoRA 的最简 mental model
原始权重 W
冻结
+
A · B
低秩可训练更新
→
W + ΔW
LoRA 在这里的好处
  • 训练参数量很小
  • 训练内存压力更可控
  • 很适合作为“整包训练流程演示”的第一版
这次真实 run 的 LoRA 核心数字

Trainable: 1,843,200 / 3,756,466,176 (0.05%)

这就是 LoRA 最值得记住的一行：训练只动了 0.05% 的参数。

为什么 target_modules 很重要：LoRA 不是自动对“整个模型”生效，而是只对配置里列出的模块生效。这次是 q_proj 和 v_proj。

6. 配置层：YAML 是怎么变成 Trainer 输入的

配置层负责回答两个问题：用户写的 YAML 是否合法？如果合法，如何变成训练器真正吃的字典结构？

src/mmit2/config/training_config.py 的职责
  • 把 YAML 解析成 dataclass：TrainingConfig
  • 校验必要字段：模型路径、数据路径、runtime mode、训练方法名
  • 为方法补默认值：例如 LoRA 的 lora_r、lora_alpha
  • 把 dataclass 再转成 training.__main__.py 期待的 dict

6.1 为什么 LoRA 校验里会强制要求 target_modules

在 training_config.py 里，LoRA 家族方法会被单独检查：如果 target_modules 为空，就直接报错。这个设计很适合教学，因为它避免了“方法自己偷偷猜目标模块”的隐式行为。

Mental model：配置层做的是“把外部世界的自由文本输入，收束成内部系统能放心消费的结构化参数”。

7. 数据层：HF dataset 如何变成 CanonicalSample

LoRA 训练真正开始前，数据层已经做了很多关键翻译工作。这里最值得盯的文件是 src/mmit2/data/adapters/hf_datasets.py。

数据层的一句话任务

把 Hugging Face 上每个数据集自己的列名与格式差异，收束成统一的 CanonicalSample 序列。

行为代码里怎么做为什么重要
选择 split尝试请求 split，必要时 fallback不同 HF 数据集不总是严格一致
决定 streaming小样本或 spec 偏好时改成 streamingColab 调试更轻
限制样本数max_samples 可直接传给 adapter适合 smoke test
解析 row依赖数据集 spec / column mapping下游不用再关心原始列名差异

8. 预处理层：一条样本如何变成训练张量

这是整份代码里最有“教学中间值”的一层。因为这一层直接把人类可读样本，翻译成模型训练真正使用的 input_ids、labels、attention_mask 和图像张量。

src/mmit2/training/preprocessors/chat_template.py 的主线
sample
  ↓
_load_image()
  ↓
_build_messages()
  ↓
processor.apply_chat_template(messages)
  ↓
full_text / prompt_text
  ↓
processor(text=..., images=..., return_tensors="pt")
  ↓
input_ids / attention_mask / pixel_values
  ↓
labels = input_ids.clone()

8.1 为什么这里要分 full_text 和 prompt_text

这是整条预处理逻辑最关键的设计之一。

  • full_text 包含完整训练对话，里面有答案。
  • prompt_text 只包含“问题加上等待回答的位置”，里面没有答案。

有了这两份文本，就能推断出：哪些 token 只是上下文，哪些 token 才是需要监督的 assistant answer。

8.2 为什么 labels[:prompt_len] = -100 这么重要

这一步不是删掉 prompt，而是说：prompt 仍然作为输入喂给模型，但不拿来计算 loss。 真正参与监督的只有 assistant answer 对应的 token。

8.3 collate 在这里做什么

tokenize() 是单样本工作；collate() 是多样本工作。它把不同长度的序列补齐，再拼成 batch。也就是：

  • batch_ids
  • batch_labels
  • batch_mask
  • batch_prompt

如果图像张量形状兼容，它也会把 pixel_values 叠起来。

最短 mental model：chat_template.py 就是在做“训练输入的翻译层”。上游是统一样本，下游是统一张量。

9. Trainer 层：谁在真正 orchestrate 整条 run

src/mmit2/training/trainer.py 不是某个训练方法本身，它更像一个总调度器。它负责决定在什么顺序里做哪些大步骤。

Trainer 的职责边界
  • 加载 model 和 processor
  • 加载并预处理数据
  • 实例化训练方法（例如 LoRA）
  • 构建 optimizer、scheduler、DataLoader
  • 执行训练循环
  • 触发 checkpoint 保存

9.1 训练循环最值得看的骨架

for epoch in range(num_epochs):
    for step, batch in enumerate(loader):
        batch = _to_device(batch, device)
        batch["labels"] = method.preprocess_labels(...)
        forward_batch = {...}  # prompt_mask 不传给 model.forward
        outputs = model(**forward_batch)
        loss, metrics = method.compute_loss(model, batch, outputs)
        loss = loss / gradient_accumulation_steps
        loss.backward()

        if should_step:
            clip_grad_norm_(...)
            optimizer.step()
            scheduler.step()
            optimizer.zero_grad()

9.2 为什么这里还要有 preprocess_labels() 钩子

因为不是所有训练方法都对 labels 的要求一样。LoRA 默认直接用预处理层给出的 labels；像 L2T 这种方法可能还要进一步修改 masking。Trainer 把这个扩展点留给了方法层。

9.3 为什么说 Trainer 是“编排层”，不是“算法层”

它关心的是流程顺序，而不是 LoRA 数学细节。LoRA 的“该挂哪些 adapter、哪些参数可训练、如何保存和恢复”都在方法层，不在 Trainer 里。

10. TrainingMethod / LoRA 层：方法真正接管了什么

如果 Trainer 是总调度器，那么 TrainingMethod 就是“可替换算法插槽”。整包通过这个抽象，把 LoRA、QLoRA、DoRA、Freeze、L2T 放在同一个接口下。

src/mmit2/training/methods/base.py（方法接口）
  • default_config()
  • requires_quantization()
  • prepare_model()
  • preprocess_labels()
  • compute_loss()
  • get_trainable_params()
  • save_checkpoint()
  • load_for_inference()

10.1 LoRA 在 lora.py 里真正做的事

  • 读取 lora_r、lora_alpha、lora_dropout、target_modules
  • 构造 peft.LoraConfig
  • 调用 get_peft_model() 把 adapter 注入到基础模型
  • 统计可训练参数量并返回信息字符串
  • 训练后用 save_pretrained() 保存 adapter
  • 推理时用 PeftModel.from_pretrained() 恢复，并尝试 merge_and_unload()

10.2 这份代码里 LoRA 的 loss 是怎么来的

LoRAMethod.compute_loss() 走的是统一的 CrossEntropyLoss helper。若 Hugging Face 模型已经在 outputs.loss 里给出 loss，就直接用；否则自己对 shift_logits 和 shift_labels 做 causal LM cross entropy。

10.3 为什么训练时显示 0.05% 可训练参数

因为绝大多数基础模型参数被冻结了，真正会被 optimizer 更新的，只有新挂上的 LoRA adapter 权重。这个数字是理解 LoRA 价值最直接的一行输出。

方法层的核心边界：Trainer 决定“训练要不要开始”；LoRA 方法决定“这个模型以什么方式变成可训练模型”。

11. 评测层：训练后的模型怎样重新拿回来做本地推理

LoRA 路线训练完之后，并不是直接拿训练中的 Python 对象就算完事。评测层要回答的是：如果我只有 checkpoint 路径，怎样把模型恢复出来并再次 generate？

    读 mmit_meta.json 确定 ft_method
    build_training_method(ft_method)
    method.load_for_inference(...)
否则:
    直接加载 base model

11.1 训练预处理和评测预处理的差别

训练时我们构造的是“问题 + 标准答案”的监督样本；评测时构造的是“问题 + 等待模型回答”的生成 prompt。所以训练看 full_text / labels，评测看的是 add_generation_prompt=True。

11.2 为什么本次 sample output 里训练前就有 0.8

因为 base model 本来就不弱。我们选的 10 条 held-out 样本也很小，而且是 smoke-test holdout，不是正式 benchmark。它更多是在回答“评测链有没有跑通”，不是在给模型能力下结论。

12A. 每一层到底收什么、产出什么

你说得对，前一版里“层与层之间是怎么接上的”还不够细。这里我们专门把每一层的 contract 摆平。这样之后看源码时，就不会只看到函数名，而是会知道：这个函数在系统里到底负责接收什么、产出什么、保证什么。

为什么这一张 contract 表很重要

因为它帮我们避免一种常见混乱：把“谁负责什么”搞丢。比如：

  • 如果你在 Trainer 里找 messages，大概率会迷路，因为那是预处理层的边界对象。
  • 如果你在 lora.py 里找 DataLoader，也会迷路，因为 batch 是 Trainer 的事，不是方法层的事。
  • 如果你在 adapter 里找 loss，也会迷路，因为数据层的责任是“翻译样本”，不是“训练模型”。

12B. Trainer.train() 逐步 walkthrough

前一版已经讲了 Trainer 的职责边界，但如果目标是“拿这份讲义去带人读源码”，那还需要再往下一层：按实际执行顺序，把 train() 真的拆开。

把 Trainer.train() 先压缩成九步
1. build_training_method(...)
2. _load_model(...)
3. _build_dataset(...)
4. _preprocess_dataset(...)
5. method.prepare_model(...)
6. method.get_trainable_params(...)
7. DataLoader(..., collate_fn=preprocessor.collate)
8. for batch in loader: forward / loss / backward / step
9. method.save_checkpoint(...)

12B.1 第一步：build_training_method()

Trainer 先根据名字从 registry 实例化方法对象。这个对象是后续整个训练 run 的“算法侧代理人”。在 LoRA 路径上，它之后会负责：

  • 方法默认配置
  • 模型 adapter 注入
  • loss 计算接口
  • 可训练参数分组
  • checkpoint 保存 / 恢复

12B.2 第二步：_load_model()

这一层调用的是 modeling.py 里的两个 helper：

  • load_processor(model_path)
  • load_vlm(model_path, quantize_4bit=..., torch_dtype=torch.bfloat16)

这里最值得记住的，不是加载细节本身，而是一个设计点：是否量化，不是 Trainer 自己决定，而是方法层通过 requires_quantization() 反向告诉 Trainer。 这就是为什么同一条训练主链可以容纳 LoRA 和 QLoRA。

12B.3 第三步：_build_dataset()

Trainer 会把 config 里的 data_config 交给 HFDatasetsAdapter。这里有几个这次调试里很重要的细节：

  • adapter 现在只支持 hf_datasets
  • data_path 会被翻成 dataset_name
  • image_root 会被剥掉，不再错误地传给 adapter
  • max_samples 直接传给 adapter，而不是先全量读再切片

这也解释了我们之前为什么会专门修 image_root 相关的 bug：那是“配置层字段”和“adapter 构造参数”之间 contract 没对齐。

12B.4 第四步：_preprocess_dataset()

这一步对每条样本做的事情其实很单纯：调用 preprocessor.tokenize(sample, processor, ...)。但它非常关键，因为从这一步之后，Trainer 手里拿到的就不再是语义层面的问答样本，而是张量层面的训练样本。

这一步之前这一步之后
CanonicalSample(id, image_path, turns, metadata){"input_ids": ..., "labels": ..., "attention_mask": ..., "prompt_mask": ..., "pixel_values": ...}

12B.5 第五步：method.prepare_model()

这是 LoRA 真正开始“动手”的地方。Trainer 把 base model 交给方法层，LoRA 则把 adapter 挂上去，并返回一段信息字符串：

LoRA: r=8, alpha=16, dropout=0.05
Target modules: ['q_proj', 'v_proj']
Trainable: 1,843,200 / 3,756,466,176 (0.05%)

从这里开始，self._model 已经不是“纯 base model”，而是“包了一层 PEFT LoRA adapter 的可训练模型”。

12B.6 第六步：optimizer 与 param groups

method.get_trainable_params(model) 返回的是参数分组，而不是单纯的参数列表。这给方法层留下了以后做差异学习率、不同模块不同 weight decay 的空间。LoRA 当前实现里，返回的是一个只有一组的简单列表，但接口本身是按扩展性设计的。

12B.7 第七步：DataLoader 为什么这样配

loader = DataLoader(
    processed,
    batch_size=config.per_device_batch_size,
    shuffle=True,
    collate_fn=preprocessor.collate,
    drop_last=True,
)

这里有三个值得特别讲的参数：

  • collate_fn=preprocessor.collate：说明 batch 的组装逻辑是预处理层提供的。
  • shuffle=True：训练样本打乱。
  • drop_last=True：最后不足一个完整 batch 的部分丢掉，这会影响总步数计算。

12B.8 第八步：为什么要先把 batch 过滤成 forward_batch

forward_batch = {
    key: value for key, value in batch.items()
    if key not in _NON_FORWARD_BATCH_KEYS
}

这里最典型被过滤掉的是 prompt_mask。原因很简单：它对训练逻辑有意义，但对 Hugging Face 模型的 forward() 没有直接参数意义。 所以它应该保留在 batch metadata 里，供方法层或调试层使用，但不应该盲目传进模型。

12B.9 第九步：loss、梯度累积和真正的 optimizer step

outputs = self._model(**forward_batch)
loss, metrics = method_obj.compute_loss(self._model, batch, outputs)
loss = loss / config.gradient_accumulation_steps
loss.backward()

这里必须单独强调一个点：backward() 并不等于“已经更新参数”。参数更新只有在后面的条件满足时才发生：

if (step + 1) % gradient_accumulation_steps == 0:
    clip_grad_norm_(...)
    optimizer.step()
    scheduler.step()
    optimizer.zero_grad()

也就是说，每个 micro-batch 先把梯度累起来，等凑够 4 个，再做 1 次真正更新。这就是为什么我们这次 30 个样本最后只有 7 个 optimizer steps。

12B.10 训练结束时到底保存了什么

Trainer 在循环里会按 save_steps 保存中间 checkpoint；结束时一定会保存一个 final 目录。真正怎么保存，交给方法层决定。LoRA 的保存逻辑是：

  • model.save_pretrained(path)
  • processor.save_pretrained(path)
  • 额外写一份 mmit_meta.json，标记 ft_method = "lora"

这就是为什么评测层之后可以只靠 checkpoint 路径恢复模型：方法名和 base model 信息已经一起落盘了。

如果只允许记住 Trainer 的一句话

Trainer 不决定“LoRA 是什么”，它决定“LoRA 这次训练 run 按什么顺序发生”。

因为脚本没有只调用 preprocessor.tokenize() 然后看输出，而是把 chat_template.py 里中间会用到的 helper 也重新显式拿出来了：

  • _build_messages()
  • _build_prompt_messages()
  • processor.apply_chat_template(...)

这样它就不只是“拿到最终张量”，而是能把“张量之前的语义中间值”也展示出来。这正是写讲义非常需要的一层。

12C.3 它为什么能知道哪些位置被监督

在 _inspect_chat_template_sample() 里，脚本显式计算了：

  • supervised_positions = (labels != -100).nonzero(...)
  • prompt_positions = prompt_mask.nonzero(...)

所以它能不仅打印“有多少监督 token”，还能准确打印“监督发生在哪几个位置”。这就是为什么我们能看到 [440, 441, 442] 这样的输出。

12C.4 它为什么要区分 inspect-only 和完整 smoke run

因为这两种模式的教学目的不一样：

  • --inspect-only：只看中间值，不加载 3B 模型，不训练，速度快，适合先理解样本流。
  • 完整 smoke run：真的训练 30 条，评测 10 条，验证整条 LoRA 路线。

这次我们之所以能先把系统讲清楚，再去跑训练，很大程度上就是因为这个模式分离做得对。

12D. 一条样本的完整旅程：从训练到评测

这里我们把同一条样本沿着系统真正走一遍。这个 section 的目标不是重复前面的定义，而是让“对象怎么变形”这件事变得非常具体。

12D.1 训练视角下，这条样本经历了哪些“身份变化”

12D.2 评测视角下，这条样本为什么又会被“重写一次”

训练时，这条样本长这样：

但评测时，我们不会把答案一起喂给模型。debug 脚本会把它改写成一个 EvalSample，问题也会追加一行 instruction：

Where is he looking?

然后再由 LocalMethod.prepare_eval_input() 把它重新包装成“只有 user 的 prompt”，交给生成模型输出答案。这也是为什么训练视角和评测视角虽然都使用 processor，但输入 contract 不同。

12D.3 为什么这一条样本最后只有 3 个监督 token

这里把前面的几个点合在一起看，就会更清楚：

  • 答案文本本身很短：down
  • 监督目标不是整条对话，而是 assistant 回答尾部
  • Qwen2.5-VL 的 tokenizer / template 会把答案尾部连同结束标记一起编码

所以最后我们看到的是：

decoded_supervised_text = 'down<|im_end|>\n'

这正是 causal LM 训练里非常常见的一种现象：上下文很长，但真正被监督的位置只是一小段尾部 token。

如果你要拿这份讲义带别人读代码，这一节最值得反复讲

因为很多人卡住，并不是卡在某个 API，而是卡在“为什么同一条数据在系统里看起来总是不一样”。这一节正好是在把这些“身份变化”讲清楚。

这是整份 sample output 里最有“教学中间值”的部分。因为它清楚地展示了：CanonicalSample → messages → rendered text → token ids → supervision mask 的完整路径。

messages 包含完整监督对话；prompt_messages 停在 assistant 回答前。这正是后面监督 masking 的来源。

15.2 full_text 与 prompt_text

'full_text':

  • prompt_text 正好停在 assistant 开始前，所以它可以用来界定监督边界。

15.3 为什么 decoded_input 里有一大串 <|image_pad|>

在 full_text 里我们看到的视觉占位像是一个简单的 <|image_pad|>；但真正经过 processor 以后，图像会被展开为更长的视觉 token 序列。所以 decoded_input 看起来会像下面这样：

<|im_start|>user
<|vision_start|><|image_pad|><|image_pad|><|image_pad|> ... <|image_pad|><|vision_end|>Where is he looking?<|im_end|>

这就是为什么本次样本的总长度会很长，即使答案本身只有一个词。

15.3A processor(...) 之后，实际上拿到了哪些 key

这一点前一版写得不够细。ChatTemplatePreprocessor.tokenize() 里真正从 processor 拿回来的，并不只是文本 token。对多模态模型来说，常见返回项包括：

key来源作用
input_ids文本与视觉占位 tokenization模型主输入；形状通常是 [1, T]
attention_maskprocessor 自动构造标记哪些位置有效，哪些是 padding
pixel_values图像预处理真正送进视觉编码器的图像张量
image_sizes部分模型会提供帮助模型理解原始图像尺度
image_grid_thwQwen2.5-VL 这类模型常见描述视觉 token 网格结构

也就是说，decoded_input 只是我们为了教学把文本侧 decode 回来看；但训练时真正 forward 进去的是一整包多模态输入，而不只是字符串 token。

15.4 这三个数字最重要

'input_length': 443
'prompt_token_count': 440
'supervised_token_count': 3
'supervised_positions_preview': [440, 441, 442]

这是整份讲义里最适合用来解释 supervision 的数字：

  • 总长度 443，不短，因为视觉 token 占了很多位。
  • 前 440 个 token 属于 prompt，不参与 loss。
  • 真正参与监督的只有最后 3 个 token，对应 down<|im_end|>。

15.5 labels_tail 最值得怎么讲

'labels_tail': [
  -100, -100, -100, ...,
  -100,
  2923,
  151645,
  198
]

这里最直观地证明了：大多数位置都被 mask 成了 -100，只有答案尾部几个 token 被保留下来作为监督目标。

15.6 这一步和 causal LM loss 是怎么接上的

到这里为止，我们已经知道哪些位置是监督目标了。但要真正算 loss，还要再过一层：causal language modeling 的“shift”规则。

shift_logits = logits[..., :-1, :]
shift_labels = labels[..., 1:]
loss = cross_entropy(
    shift_logits.view(-1, vocab_size),
    shift_labels.view(-1),
    ignore_index=-100,
)

它的意思是：

  • 位置 t 的 logits，要去预测位置 t+1 的 label。
  • 所以即便我们把 labels = input_ids.clone()，也不是在让模型“抄答案”，而是在让模型用前面的 token 去预测后面的 token。
  • 被标成 -100 的位置在 cross-entropy 里会被忽略，所以 prompt 不参与训练损失。

把这一步和前面的 supervised_positions = [440, 441, 442] 放在一起看，就很容易理解：模型真正被要求学的，是“在看完前面的上下文后，如何继续生成 down<|im_end|> 这一小段尾部 token”。

16.2A loss 在 LoRA 路线里是谁算的

这也是一个前一版只提到但没有讲透的点。LoRA 路线里，compute_loss() 最终走的是统一的 CrossEntropyLoss helper：

  • 如果 Hugging Face 模型 forward 后直接给了 outputs.loss，就直接用。
  • 如果没有，就自己从 outputs.logits 和 batch["labels"] 重新算一遍 causal cross-entropy。

这个设计很好地把“LoRA 是什么”和“loss 怎么算”解耦开了：LoRA 负责 adapter 注入，loss helper 负责语言建模监督。
