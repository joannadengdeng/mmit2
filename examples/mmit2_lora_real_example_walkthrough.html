Step 3. 加载 train / eval slice：为什么日志里会反复看到 Resolving data files

现在脚本开始显式加载数据。这里最容易让人误会的是：为什么一份小数据，输出里会看到这么多次 Hugging Face 的数据解析日志？

3.1 这次脚本其实加载了三次数据

  1. _load_samples()：加载训练用的前 30 条，只为了调试和展示。
  2. _load_eval_samples()：再加载前 40 条，然后切出后 10 条做 held-out eval。
  3. Trainer.train() 内部：训练真正开始时，再正式加载一次训练样本。

所以你在 Colab 输出里看到多次 Resolving data files 是合理的。这份 debug harness 的设计目标是“可讲解”，不是“最省数据读取次数”。

3.2 _build_adapter_kwargs() 把配置怎么翻成 adapter 参数

脚本会先把 data config 做一次翻译：

  • 取出 adapter，并强制要求它是 hf_datasets
  • 把 data_path 改名为 dataset_name
  • 把 image_root 从 adapter 参数里剥掉
  • 根据不同场景决定 max_samples

也就是说，真正传给 HFDatasetsAdapter 的，大致是：

dataset_name = "lmms-lab/VQAv2"
split = "train"
max_samples = 30              # train sample load
max_samples = 40              # eval slice load (offset 30 + count 10)

3.3 为什么这里会走 streaming

这是一个很适合讲清楚的真实细节。VQAv2Spec 里写了：

dataset_name = "lmms-lab/VQAv2"
prefer_streaming = True

而 HFDatasetsAdapter 又有一条逻辑：

if not streaming and (prefer_streaming or (max_samples is not None and max_samples <= 50)):
    streaming = True

这次我们既满足 prefer_streaming=True，又满足 max_samples <= 50，所以小样本调试时自然走 streaming。这也是为什么之前会暴露出 streaming 下 __len__ 的特殊问题。

3.4 streaming 下，这个 adapter 的真实行为是什么

HFDatasetsAdapter 在 streaming 模式下有两条非常关键的行为：

def __len__(self):
    if self.streaming:
        return self.max_samples if self.max_samples is not None else -1

def __iter__(self):
    count = 0
    for idx, row in enumerate(self._hf_dataset):
        if self.max_samples is not None and count >= self.max_samples:
            break
        yield self._spec.parse_row(row, idx, load_images=self.load_images)
        count += 1

这说明：

  • 长度并不是总能从数据源真实拿到，所以 streaming 下常常只是“上界”或未知值。
  • 真正决定停在第几条的，是 __iter__() 里的计数逻辑。

也正因为如此，debug 脚本后面不再用脆弱的 list(adapter) 假定长度行为，而是显式按迭代收集样本。

4.1 VQAv2 的 spec 明确写了怎么取字段

class VQAv2Spec(HFDatasetSpec):

4.2 parse_row() 对这一条 row 做了什么

  1. 从 question_id 取出 id，得到 "262148000"。
  2. 从 question 取出问题，得到 "Where is he looking?"。
  3. 从 multiple_choice_answer 取出答案，得到 "down"。
  4. 从 image 字段解析图片；因为 HF 已经把图像解码成 PIL image，所以结果是 image_path = "<in_memory>"，并把真实图像对象塞进 metadata["_pil_image"]。
  5. 把问题和答案分别包成两个 Turn。

4.2A handle_image_value() 为什么要这么写

图像字段在 Hugging Face 数据集里并不总是同一种格式，所以这段 helper 本质上是在做“图像字段归一化”。

image 字段原始形态函数会怎么处理这次 run 属于哪一种
字符串路径image_path = that string不是
带 bytes / path 的 dict能解码时生成 _pil_image，否则保留延迟路径有可能，但这次不是最终表现
PIL.Image 对象metadata["_pil_image"] = image，image_path = "<in_memory>"是

这也是为什么整包明明统一要求样本里有 image_path，但我们仍然能优雅地处理“图像已经在内存里”的情况。

_build_prompt_messages() 的逻辑是：找到最后一个 assistant turn，并把它以及它后面的内容截掉。对这条两轮样本来说，结果就是只留下第一条 user message：

之后它会被用来生成 prompt_text，也就是“问题已经给出，但答案还没出现”的版本。

6.2A 这两段 helper 的源码到底在表达什么

def _build_messages(sample, has_image):
    for turn in sample.turns:
        role = "user" if turn.role == "human" else "assistant"
        if role == "user" and has_image and not messages:
            content = [{"type": "image"}, {"type": "text", "text": turn.content}]
        else:
            content = [{"type": "text", "text": turn.content}]

def _build_prompt_messages(messages):
    find last assistant index
    return messages before that assistant turn

这里其实有两个非常具体的设计选择：

  • 图像只会被挂在第一条 user turn 里，而不是后面每轮都重复挂。
  • prompt_messages 的定义不是“只保留 user”，而是“保留到最后一个 assistant 之前为止”。这在多轮对话里尤其重要。

7.3 为什么 decoded_input 里会有很长的 <|image_pad|> 串

debug 输出里你看到的长串视觉 token，大概会长这样：

<|vision_start|><|image_pad|><|image_pad|><|image_pad|> ... <|image_pad|><|vision_end|>

这说明一件很重要的事：对多模态模型来说，图像不是只占一个特殊 token，而是会展开成一段更长的视觉 token 序列。 这就是为什么答案只有一个词，整条输入却能有 443 个 token。

7.4 labels 到底是怎么构造的

labels = input_ids.clone()
labels[:prompt_len] = -100

这一步看起来简单，但它是整条训练监督的核心。它的意思不是删掉 prompt，而是：

  • prompt 仍然作为输入喂给模型。
  • 但 prompt 对应的位置不参与 loss。
  • 只有 assistant answer 那一小段 token 会被监督。
'labels_tail': [
  -100, -100, -100, ...,
  -100,
  2923,
  151645,
  198
]

这段尾部输出最能直观说明问题：大量位置被 mask 成 -100，只有最后三个 token 被保留下来。

7.5 为什么这样就能训练语言模型

这里要补一个 PyTorch / causal LM 基础：loss 不是直接拿同一个位置的 logits[t] 去对比 labels[t]，而是做一个“向后错一位”的 shift：

shift_logits = logits[..., :-1, :]
shift_labels = labels[..., 1:]
loss = cross_entropy(
  shift_logits.view(-1, vocab_size),
  shift_labels.view(-1),
  ignore_index=-100,
)

所以即使 labels = input_ids.clone()，模型学的仍然是“根据前面的 token 去预测后面的 token”。在我们的样本里，它真正要学的是：看完问题和图像后，怎样继续生成 down<|im_end|> 这几个 token。

这一步是整份讲义最值得彻底吃透的地方

如果你能把 CanonicalSample → messages → full_text / prompt_text → input_ids / labels / prompt_mask 这一整段顺清楚，整包的很多“魔法感”都会消失。

Step 12. DataLoader 与 7 个优化步：这次真实训练到底怎么数出来的

这一节是例子版讲义最值得细讲的地方之一。因为“为什么只有 7 步”其实能把 DataLoader、micro-batch、梯度累积和 optimizer step 全串起来。

12.1 先把真实配置列出来

num_epochs = 1
per_device_batch_size = 1
gradient_accumulation_steps = 4
train_sample_count = 30
drop_last = True

因此：

  • DataLoader 里一共有 30 个 micro-batches。
  • 每 4 个 micro-batches 才做 1 次真正的 optimizer step。

12.2 Trainer 里是怎么计算 total_steps 的

steps_per_epoch = max(1, len(loader) // gradient_accumulation_steps)
total_steps = steps_per_epoch * num_epochs

代入真实数字就是：

len(loader) = 30
gradient_accumulation_steps = 4
steps_per_epoch = 30 // 4 = 7
total_steps = 7 * 1 = 7

12.3 这 7 步具体发生在哪些 micro-batches

因为 Trainer 的更新条件是：

if (step + 1) % gradient_accumulation_steps == 0:
    optimizer.step()
    scheduler.step()
    optimizer.zero_grad()

所以真正发生更新的是 micro-batch：

4, 8, 12, 16, 20, 24, 28

这正好对应 7 次 optimizer step。

12.3A 如果按 micro-batch 分组，会长成什么样

micro-batch 范围发生了什么对应 optimizer step
1-44 次 forward / loss / backward，累积梯度step 1
5-8再次累积 4 次step 2
9-12再次累积 4 次step 3
13-16再次累积 4 次step 4
17-20再次累积 4 次step 5
21-24再次累积 4 次step 6
25-28再次累积 4 次step 7
29-30会做 backward，但 epoch 结束前不再触发 optimizer.step()无

这张表特别适合拿来讲“梯度累积”这个概念，因为它把抽象的 gradient_accumulation_steps = 4 变成了非常具体的事件流。

12.4 一个很值得坦白写进讲义的真实细节

30 不能被 4 整除，所以第 29 和 30 个 micro-batches 会做 backward()，但不会触发一次新的 optimizer.step()。也就是说，这份实现当前不会在 epoch 结束时“冲刷掉”最后不完整的一组梯度累积。

这不影响这次 smoke run 跑通，但它是一个很真实的实现细节。 如果以后你要把这份讲义拿去做深入代码讲解，这一行非常值得单独指出，因为它正好能帮助读者区分“micro-batch 次数”和“真正参数更新次数”。

12.5 学习率为什么从第一步开始就在下降

配置里虽然写了 warmup_ratio = 0.03，但 Trainer 实际是：

warmup_steps = int(total_steps * warmup_ratio)

代入这次 run：

warmup_steps = int(7 * 0.03) = int(0.21) = 0

所以这次实际上没有 warmup，scheduler 从一开始就走 cosine 下降。这也解释了为什么第一步记录到的 lr 已经是：

0.0001900968867902419

而不是初始设定的 0.0002。

关于整包结构
  • mmit2 的核心不是某一个“神秘训练函数”，而是一条非常明确的对象流：YAML → CanonicalSample → messages → full_text → tensors → LoRA model → checkpoint → LocalMethod eval。
  • 数据层、预处理层、方法层、Trainer 层、评测层的边界是清晰的；每一层都在“翻译对象”，而不是随意混杂。
关于这次 LoRA 路线
  • LoRA 真正做的是：在指定模块上挂 adapter，并让 0.05% 的参数可训练。
  • 训练前后分数没变，但这次 run 已经完整验证了训练和评测主链。
  • LoRA 训练时和训练后推理时的模型加载路径并不相同，所以某些 summary 不能机械对比。
关于中间输出
  • One Raw Sample 让我们看到统一样本长什么样。
  • messages / full_text / prompt_text 让我们看到 chat template 层的语义重写。
  • input_length = 443, prompt_token_count = 440, supervised_token_count = 3 让我们看到监督边界如何真正编码进 labels。
  • Trainable: 0.05% 和 7 个 optimizer steps 让我们看到 LoRA 训练主链真的活着。
