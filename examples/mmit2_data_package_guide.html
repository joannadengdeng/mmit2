mmit2 data 包讲义：从 HF row 到 CanonicalSample

这份讲义专门讲 src/mmit2/data。重点不是“把每个文件逐个念一遍”，而是先建立 mental model，再沿着一条真实 VQAv2 样本的流动过程，讲清 HF dataset row → dataset spec → adapter → CanonicalSample → Trainer / Eval 这条主线。
读法建议：如果你要教别人这部分代码，最值得反复强调的是一句话：data 包的目标不是“把数据集原样暴露出来”，而是“把各种来源的数据重写成 mmit2 自己的标准语言”。 这个标准语言就是 CanonicalSample 和 EvalSample。

这份讲义在做什么

很多人一看到 src/mmit2/data，会先把注意力放在“支持了哪些数据集”上。但工程上更关键的问题其实是：这些数据集被转换成了什么样的统一对象，后面的训练和评测为什么能不关心原始格式差异。

一句话目标

mmit2 的 data 包，本质上是在做“数据格式归一化”。 它把不同 HF 数据集的列名、图像字段、答案字段、是否 streaming、是否内存解码，最后都收束成统一的 CanonicalSample。

你读这份讲义时要一直记住的中心句

训练层并不想知道你原始数据集叫不叫 question_id、图像是 bytes 还是 path、答案是单字符串还是 list。 data 包的职责，就是在进入 Trainer 之前把这些差异吃掉。

当前边界

讲 data 包时，边界一定要讲清楚。不然读者会以为架构已经比实际交付得更广。

  • 当前训练 config 的 data.adapter 只支持 hf_datasets。
  • 当前 registry 内建的 dataset spec 只有三个：lmms-lab/VQAv2、lmms-lab/textvqa、lmms-lab/VizWiz-VQA。
  • 虽然 HFDatasetsAdapter 支持传入 column_map，但 YAML config 目前没有把这条能力公开成正式配置项。
  • image_root 在 HF 数据集路径里通常不是核心角色，因为图像常常已经在 row 里；但 preprocessor 和 eval 仍然保留了它，兼容“文件路径型图像”。
# training_config.py 的验证逻辑
if cfg.data.adapter != "hf_datasets":
    errors.append(
        f"data.adapter: '{cfg.data.adapter}' is not supported. "
        "Only 'hf_datasets' is supported."
    )
这点在教学里一定要点明

架构上是“adapter 可扩展”，产品上是“当前只正式支持 hf_datasets”。 这两句话同时成立，而且都要讲。

1. Turn：最小对话单位

先从最小粒度开始。types.py 里的 Turn 很简单，但它决定了后面训练数据是怎么被想象成“对话”的。

class Turn:
    role: str     # "human" | "assistant"
    content: str
  • role 在 data 层用的是 "human" / "assistant"。
  • 到了 preprocessor 的 chat-template 层，会被改写成 Hugging Face 更常见的 "user" / "assistant"。
  • 也就是说，data 层的 role 命名不必和模型模板完全一致，只要后面的桥接层知道怎么转换就行。
很容易忽略的一点：Turn 的存在说明 mmit2 在 data 层就已经把样本看成“多轮对话容器”，哪怕当前大多数 VQA 训练样本实际只有一问一答。

2. CanonicalSample：训练路径的标准语言

这是 data 包最核心的对象。你可以把它当成“训练世界里的 lingua franca”。

class CanonicalSample:
    id: str
    image_path: str
    turns: List[Turn]
    metadata: Dict[str, Any] = field(default_factory=dict)

    @property
    def first_question(self) -> str: ...

    @property
    def first_answer(self) -> str: ...
这四个字段各自承担什么角色
  • id：样本身份标识，后面 debug / eval / report 都会打印它。
  • image_path：图像来源描述；可以是真路径，也可以是特殊占位符。
  • turns：真正的语义主体。
  • metadata：保留不适合塞进主 schema 的附加信息，例如 raw_answers、_pil_image。
为什么要有 first_question / first_answer
  • 它们不是为了表达完整语义，而是为了给 debug / eval / summary 快速取第一问第一答。
  • 这是一种很实用的“展示层 convenience API”。
  • 后面的 debug 脚本大量依赖它们打印可读摘要。
# 真实例子
CanonicalSample(
  id="262148000",
  image_path="<in_memory>",
  turns=[
    Turn(role="human", content="Where is he looking?"),
    Turn(role="assistant", content="down"),
  ],
  metadata={
    "_pil_image": <PIL image ...>,
    "raw_answers": ["down"],
  },
)
这一层的核心设计判断

CanonicalSample 故意不强绑定任何具体 benchmark schema。 它不要求一定有 question_id、也不要求答案一定叫 multiple_choice_answer。它只关心：这条样本最后能不能用“图像 + 一串对话 turn”表达出来。

3. EvalSample：评测路径的标准语言

EvalSample 和 CanonicalSample 长得像，但使用场景不同。讲这两个对象时一定要明确区分训练路径和评测路径。

class EvalSample:
    id: str
    image_path: str
    question: str
    ground_truth: Optional[Any] = None
    metadata: Dict[str, Any] = field(default_factory=dict)
  • CanonicalSample 面向训练或通用多轮输入。
  • EvalSample 面向 benchmark question，一般是一问一答型评测单元。
  • ground_truth 的类型故意放得比较宽，因为有时是 VQA 多答案 list，有时可能是 MCQ 选项。
# eval/methods/base.py
def prepare_eval_input(self, sample: EvalSample, image_root: str = "") -> Dict[str, Any]:
    cs = CanonicalSample(
        id=sample.id,
        image_path=sample.image_path,
        turns=[Turn(role="human", content=sample.question)],
        metadata=sample.metadata,
    )
    return self.prepare_input(cs, image_root=image_root)
很值得讲清的一点：评测路径并没有发明一套新的 tokenizer 逻辑；它是先把 EvalSample 临时包装回 CanonicalSample，然后复用同一套 prepare / template / generate 管线。

4. 为什么要有 dataset spec

data 包里最容易被低估的设计，就是把“如何解释某个 HF row”抽成了一层 spec，而不是直接把解析逻辑写死在 adapter 里。

spec 层回答的是“这一行是什么意思”

同样是一个 Hugging Face dataset row，不同数据集的列名完全可能不同：

  • 有的 id 叫 question_id
  • 有的答案是 multiple_choice_answer
  • 有的答案是 answers list
  • 有的图像已经是 PIL image，有的是 bytes/path 结构

spec 层就是负责把这些差异翻译成同一套规则。

HFDatasetsAdapter ├─ 负责 load_dataset(...) ├─ 负责 streaming / slicing / iteration └─ 不想知道 “question 字段到底叫什么” HFDatasetSpec ├─ 负责 column mapping ├─ 负责 parse_question / parse_answer ├─ 负责 build_metadata └─ 负责 parse_row() → CanonicalSample

5. ColumnMapping：最基础的 schema 翻译表

ColumnMapping 是最轻量的一层。它做的不是解析，而是先告诉系统“哪个列名对应哪个语义”。

class ColumnMapping:
    id_col: str = "id"
    image_col: str = "image"
    question_col: str = "question"
    answer_col: str = "answer"
字段含义例子
id_col样本 ID 来自哪一列question_id
image_col图像字段来自哪一列image
question_col问题来自哪一列question
answer_col答案来自哪一列multiple_choice_answer 或 answers
这一层不要过度神化：ColumnMapping 只是“列名映射表”，它本身不处理 list 答案、也不处理图像 bytes。真正的解析逻辑在 HFDatasetSpec 里。

6. handle_image_value()：图像字段的翻译器

这是 data 包里一个非常关键的小函数，因为多模态训练里最麻烦的差异之一就是“图像这一列到底长什么样”。

def handle_image_value(image_val, load_images: bool = True) -> Tuple[str, Dict[str, Any]]:
    metadata = {}
    image_path = ""

    if image_val is None:
        pass
    elif isinstance(image_val, str):
        image_path = image_val
    elif isinstance(image_val, dict) and ("bytes" in image_val or "path" in image_val):
        ...
    else:
        ...

    return image_path, metadata
它可能收到的 image_val 形态
  • None
  • 字符串路径
  • HF Image 的 dict 形式：带 bytes / path
  • 已经 decode 完的 PIL.Image
它的返回值为什么是二元组
  • image_path：给主 schema 一个统一位置
  • metadata：把实际图像对象或 bytes 放到旁路
  • 这样上层拿到的 CanonicalSample 就保持稳定
输入情况load_images=True 时结果为什么这么设计
字符串路径image_path=that_string最直观，后面可配合 image_root 读文件
dict 且有 bytes尝试解码为 PIL，metadata["_pil_image"]，image_path="<in_memory>"说明图像已经在内存里，不必再走文件系统
dict 但不解码image_path=path 或 "<deferred>"，可能存 _image_bytes允许推迟图像解码
PIL.Imagemetadata["_pil_image"]=image，image_path="<in_memory>"显式告诉后面“这张图已经在内存中”
# 这次真实样本就是这一路
image_path = "<in_memory>"
metadata["_pil_image"] = <PIL.JpegImagePlugin.JpegImageFile image mode=RGB size=640x512 ...>
教程里一定要解释清楚 “<in_memory>”

它不是一个真路径，而是一个语义标记：“图像已经在 metadata 里了，别再去文件系统找。” 这也是为什么后面的 preprocessor 会优先看 metadata["_pil_image"]。

7. HFDatasetSpec：一行 row 如何变成 CanonicalSample

HFDatasetSpec 是整个 datasets 层的核心。它不是某个具体数据集，而是“HF VQA 数据集 spec 的共同行为模板”。

class HFDatasetSpec:
    dataset_name: str = ""
    mapping: ColumnMapping = ColumnMapping()
    prefer_streaming: bool = False

    def parse_question(self, row: dict) -> str: ...
    def parse_answer(self, row: dict) -> tuple[str, Any]: ...
    def build_metadata(self, answer_raw: Any) -> Dict[str, Any]: ...
    def parse_row(self, row: dict, idx: int, load_images: bool = True) -> CanonicalSample: ...

7.1 parse_question()

它做的事很朴素：按照 mapping 取 question 列，如果拿到 list，就取第一个；最后都收束成字符串。

7.2 parse_answer()

这里开始有一点语义判断了：

  • 如果答案是 list，而且 list 里还是 dict，那么默认取第一个 dict 里的 answer 作为训练时的主答案。
  • 但它并不会丢掉原始答案集合，而是把原始对象作为 answer_raw 返回给后面的 build_metadata()。

7.3 build_metadata()

这一步专门负责把“原始答案集合”保留到 metadata["raw_answers"]。这对 VQA 很重要，因为训练时可能只监督一个答案，但评测时经常需要保留多个参考答案。

7.4 parse_row()

这是整层的关键函数。它会：

  1. 解析 question
  2. 解析主答案和原始答案
  3. 构建 Turn("human", question) 和 Turn("assistant", answer)
  4. 调用 handle_image_value() 处理图像列
  5. 合并图像 metadata 和答案 metadata
  6. 返回完整 CanonicalSample
return CanonicalSample(
    id=sample_id,
    image_path=image_path,
    turns=turns,
    metadata=metadata,
)
设计味道很对的一点：parse_row() 不关心数据是 streaming 还是本地，也不关心 DataLoader；它只负责“把一行 row 翻译成统一对象”。这是个很干净的职责边界。

8. 内建 spec：VQAv2 / TextVQA / VizWiz

内建 spec 本身很轻，因为大部分共享逻辑已经在 HFDatasetSpec 里了。每个具体 spec 主要做两件事：填 dataset_name，填 column mapping。

class VQAv2Spec(HFDatasetSpec):
    dataset_name = "lmms-lab/VQAv2"
    mapping = ColumnMapping(
        id_col="question_id",
        image_col="image",
        question_col="question",
        answer_col="multiple_choice_answer",
    )
    prefer_streaming = True
class TextVQASpec(HFDatasetSpec):
    dataset_name = "lmms-lab/textvqa"
    mapping = ColumnMapping(
        id_col="question_id",
        image_col="image",
        question_col="question",
        answer_col="answers",
    )
class VizWizVQASpec(HFDatasetSpec):
    dataset_name = "lmms-lab/VizWiz-VQA"
    mapping = ColumnMapping(
        id_col="question_id",
        image_col="image",
        question_col="question",
        answer_col="answers",
    )
数据集关键差异对 parse_answer() 有什么影响
VQAv2答案列是单字符串 multiple_choice_answer主答案直接就是该字符串
TextVQA答案列是 answers list主答案取第一个，同时把全部答案写入 raw_answers
VizWiz也常是 answers list和 TextVQA 类似
VQAv2Spec 里最值得单独讲的一行

prefer_streaming = True。这不是解析层的语义，而是给 adapter 的性能/加载策略一个提示：这个数据集更倾向于 streaming 方式读取。

9. registry.py：dataset_name 怎么找到 spec

registry 非常轻，但在 mental model 里很重要，因为它是从“字符串名字”跳到“具体 parser 行为”的桥。

_SPEC_CLASSES = (
    VQAv2Spec,
    TextVQASpec,
    VizWizVQASpec,
)

DATASET_SPECS = {
    spec_cls.dataset_name: spec_cls()
    for spec_cls in _SPEC_CLASSES
}
  • get_dataset_spec(dataset_name)：如果是内建数据集，就取现成 spec。
  • build_configured_spec(dataset_name, mapping)：如果你手里有自定义 column_map，可以动态构造一个 ConfiguredVQASpec。
这里要讲清一个现实点：运行时虽然有 build_configured_spec() 这条能力，但当前 YAML config 没直接暴露它，所以它更像“面向程序化扩展”的能力，而不是面向最终用户的标准入口。

10. DatasetAdapter：adapter 层的最小契约

adapter 抽象层的设计很克制。它不规定太多，只要求两件事：能迭代，能给长度。

class DatasetAdapter(ABC):
    @abstractmethod
    def __iter__(self) -> Iterator[CanonicalSample]: ...

    @abstractmethod
    def __len__(self) -> int: ...
  • 这说明 data 层输出的核心接口是“一个可迭代的 CanonicalSample 序列”。
  • 它没有强制必须支持随机访问，也没有强制必须是 torch Dataset。
  • 真正的随机访问能力是在具体实现里再决定，比如 HFDatasetsAdapter.__getitem__() 只在非 streaming 模式下可用。
这层设计的好处

如果未来你想接 CSV、JSONL、数据库、对象存储，不必先把它塞进 Hugging Face Dataset 对象，只要最后能持续 yield CanonicalSample 就行。

11. HFDatasetsAdapter：当前版本最重要的现实入口

虽然 adapter 架构是开放的，但当前真正承担大部分训练数据入口责任的是 HFDatasetsAdapter。

class HFDatasetsAdapter(DatasetAdapter):
    def __init__(
        self,
        dataset_name: str,
        split: str = "train",
        column_map: Optional[ColumnMapping] = None,
        max_samples: Optional[int] = None,
        streaming: bool = False,
        trust_remote_code: bool = True,
        load_images: bool = True,
        config_name: Optional[str] = None,
    ) -> None:
        ...
构造函数最关键的输入
  • dataset_name：HF 数据集名
  • split：想读哪个 split
  • max_samples：小样本 smoke / debug 时非常关键
  • streaming：是否用 iterable 方式读
  • load_images：是否急着把图像解码成 PIL
构造函数里实际做了什么
  • 根据 dataset_name 取 spec
  • 决定是否偏向 streaming
  • 调用 _load_dataset() 真正去拿 HF dataset
  • 必要时关闭 eager image decode
  • 必要时对非 streaming dataset 做 select 截断
  • 决定最终使用哪个 spec
spec = get_dataset_spec(dataset_name)
prefer_streaming = spec.prefer_streaming if spec is not None else False

if not streaming and (prefer_streaming or (max_samples is not None and max_samples <= 50)):
    streaming = True
    self.streaming = True
这几行特别值得讲：small-sample 调试和某些数据集的默认建议，都会把 adapter 自动推向 streaming 模式。这就是为什么你在 Colab debug 里会频繁看到 streaming 行为。

12. _load_dataset()：split fallback 与 streaming fallback

这段逻辑是 adapter 真正体现“工程味”的地方。它不是单纯一把梭 `load_dataset(name, split=...)`，而是尽量容错。

  1. 先尝试用 load_dataset_builder(...).info 看有哪些 split。
  2. 如果用户指定的 split 不存在，就尝试使用 builder 暴露的第一个 split。
  3. 无论如何，再补上 train / validation / test 这几个常见 fallback。
  4. 对每个 candidate split，先尝试普通 load，再尝试带 trust_remote_code 的 load。
  5. 如果非 streaming 路径都失败，再退回 streaming load 试一次。
for try_split in splits_to_try:
    load_kwargs = {"split": try_split, "streaming": streaming}
    for kwargs in (load_kwargs, {**load_kwargs, "trust_remote_code": trust_remote_code}):
        try:
            dataset = datasets_mod.load_dataset(*load_pos, **kwargs)
            self.split = try_split
            return dataset
        except Exception as exc:
            ...
它在工程上解决了什么问题

用户配置里写了一个 split，不代表目标数据集真的有那个 split，也不代表在当前环境下非 streaming 一定能成功。_load_dataset() 的容错逻辑，就是把这些“理想输入”变成“尽量成功的实际加载”。

13. streaming / max_samples / __len__：这层最容易踩坑的地方

这是 data 包里非常值得单独讲的一个工程点，因为 streaming dataset 在 Python 容器语义上和普通 list-like dataset 并不一样。

def __len__(self) -> int:
    if self.streaming:
        return self.max_samples if self.max_samples is not None else -1
    return len(self._hf_dataset)
  • 非 streaming 时，长度就是 HF dataset 的真实长度。
  • streaming 时，如果已经给了 max_samples，那长度就直接返回这个上限。
  • 如果 streaming 且没有上限，长度返回 -1，表示“未知或不适合当常规有限序列看待”。
这里的隐含工程约束：调用方如果想安全地把 streaming adapter 收成 list，最好先给 max_samples。这也是为什么 debug / smoke 路径都很依赖小样本上限。
def __iter__(self) -> Iterator[CanonicalSample]:
    count = 0
    for idx, row in enumerate(self._hf_dataset):
        if self.max_samples is not None and count >= self.max_samples:
            break
        yield self._spec.parse_row(row, idx, load_images=self.load_images)
        count += 1
这段逻辑的真正意义

max_samples 不只是“训练时偷懒少读一点”，它还是 streaming 语义下把“无限/未知长度可迭代对象”重新拉回可控调试范围的关键工具。

14. __iter__ / __getitem__ / profile：调用方实际能用什么

一个接口设计好不好，最终要看上层怎么实际使用它。

def __getitem__(self, idx: int) -> CanonicalSample:
    if self.streaming:
        raise TypeError("__getitem__ is not supported in streaming mode.")
    row = self._hf_dataset[idx]
    return self._spec.parse_row(row, idx, load_images=self.load_images)
  • __iter__：主路径；Trainer 和 debug 都主要靠它。
  • __getitem__：只在非 streaming 模式下可用，方便随机抽样。
  • column_names：暴露底层列名，便于调试。
  • mapping：暴露当前 spec 的列映射。
  • profile：暴露完整 spec 对象，便于教学或高级检查。
这里很适合讲一个 API 设计判断：mmit2 没把底层 HF dataset 完全藏起来，而是适度暴露了 column_names 和 profile 这种“对调试很有用，但又不会破坏抽象层”的信息。

15. 一条真实 VQAv2 row：从原始字段开始看

现在我们把上面的抽象层全部落到真实例子上。我们固定这条训练样本：

sample_id = "262148000"
question = "Where is he looking?"
answer = "down"
image_info = {mode: "RGB", size: (640, 512)}

对于 VQAv2Spec，最重要的列映射是：

ColumnMapping(
  id_col="question_id",
  image_col="image",
  question_col="question",
  answer_col="multiple_choice_answer",
)

也就是说，原始 HF row 至少要被看成下面这样的语义对象：

row = {
  "question_id": 262148000,
  "image": <HF image field / PIL image / bytes+path dict>,
  "question": "Where is he looking?",
  "multiple_choice_answer": "down",
}
这一步最重要的不是字面字段，而是“spec 认为哪一列是什么”。 同一个 row，如果换一个 spec 解释，最后出来的 CanonicalSample 就可能完全不同。

这是整份讲义最应该反复看的例子。

17. Trainer 怎么接住 CanonicalSample

data 包真正的价值，只有在上层不用再关心原始 schema 时，才算兑现。Trainer 就是最好的验证点。

# trainer.py
data_cfg = dict(config.data_config)
adapter_name = data_cfg.pop("adapter", "hf_datasets")
max_samples = int(data_cfg.pop("max_samples", 0) or 0)
data_cfg.pop("image_root", None)

adapter = HFDatasetsAdapter(
    max_samples=max_samples if max_samples > 0 else None,
    **data_cfg,
)
samples = list(adapter)
  • Trainer 只关心从 config 提取出 adapter 参数。
  • 一旦 adapter 构造完成，Trainer 拿到的就是 samples = list(adapter)。
  • Trainer 不需要知道 question 列名，不需要知道答案是 list 还是 string。
很值得讲的一个小点：image_root 在这里被显式 pop 掉，因为它不是 adapter 的参数，而是 preprocessor / eval 在真正读图片文件时才需要的参数。

18. Preprocessor 怎么接住它

一旦样本进入 preprocessor，data 包的职责就基本完成了。但如果不看这一站，你就很难真正理解 data 包为什么长成现在这样。

# chat_template.py
def _load_image(sample: CanonicalSample, image_root: str = ""):
    pil_image = sample.metadata.get("_pil_image")
    if pil_image is not None:
        return pil_image.convert("RGB")
    img_path = os.path.join(image_root, sample.image_path) if image_root else sample.image_path
    ...

这段代码正好解释了为什么 data 包要把图像处理成“主 schema + metadata 旁路”两条线：

  • 如果图像已经在 metadata["_pil_image"]，那就直接用它。
  • 如果没有，才退回 image_root + image_path 去文件系统找图。
# 真实例子接下来的 messages
[
  {
    "role": "user",
    "content": [
      {"type": "image"},
      {"type": "text", "text": "Where is he looking?"}
    ]
  },
  {
    "role": "assistant",
    "content": [
      {"type": "text", "text": "down"}
    ]
  }
]
data 包设计为什么是对的

如果 data 层只存一条僵硬的磁盘路径，HF in-memory 图像这条路径就会变得很别扭。现在这套设计让 preprocessor 可以自然地优先吃内存图像，再 fallback 到文件路径。

19. EvalSample 怎么反向接回训练世界

评测路径看似和训练路径不同，但它最后仍然会借助 data 包定义的对象语言。

# eval/benchmarks/vqav2.py
yield EvalSample(
    id=str(q["question_id"]),
    image_path=os.path.join(self.image_root, q["image"]) if self.image_root else q["image"],
    question=q["text"],
    ground_truth=self._gt_map.get(str(q["question_id"]), []),
)

然后在 Method.prepare_eval_input() 里，又会被重新包成：

CanonicalSample(
  id=sample.id,
  image_path=sample.image_path,
  turns=[Turn(role="human", content=sample.question)],
  metadata=sample.metadata,
)
这说明什么：CanonicalSample 不只是“训练数据对象”，它其实还是整包里更底层、更通用的“模型输入前对象语言”。EvalSample 只是 benchmark 侧更顺手的包装。

20. config 层如何约束 data

data 包不是孤立存在的。用户真正触发它，通常是从 YAML config 开始。

class DataConfig:
    adapter: str = "hf_datasets"
    data_path: str = ""
    split: str = "train"
    image_root: str = ""
    max_samples: int = 0
data_config = {
    "adapter": cfg.data.adapter,
    "data_path": cfg.data.data_path,
    "split": cfg.data.split,
    "image_root": cfg.data.image_root,
}
if cfg.data.max_samples:
    data_config["max_samples"] = cfg.data.max_samples
  • data_path 最后会被 Trainer 转成 adapter 侧的 dataset_name。
  • image_root 虽然保留在 config 里，但不是 adapter 主体参数。
  • max_samples 对调试、smoke、streaming 的行为影响很大。
讲给用户时可以这样概括

用户只要在 YAML 里说清“数据来自哪个 HF 数据集、读哪个 split、最多拿多少条”，data 包就负责把它翻译成训练代码真正需要的标准对象。

21. 怎么加一个新数据集

如果你的目标还是 Hugging Face datasets，只是 schema 不同，那大多数情况下你不需要写新 adapter，只需要加一个新的 spec。

  1. 在 src/mmit2/data/datasets/ 新建一个 spec 文件，例如 myset.py。
  2. 定义一个继承 HFDatasetSpec 的类，填 dataset_name 和 mapping。
  3. 如果 question / answer 解析逻辑不一样，重写 parse_question() 或 parse_answer()。
  4. 如果 metadata 需要额外字段，重写 build_metadata()。
  5. 在 registry.py 的 _SPEC_CLASSES 里注册它。
class MyDatasetSpec(HFDatasetSpec):
    dataset_name = "my-org/my-vqa"
    mapping = ColumnMapping(
        id_col="qid",
        image_col="image",
        question_col="prompt",
        answer_col="answers",
    )
经验判断：只要底层还是 HF dataset row，优先加 spec，不要急着加 adapter。adapter 是为“数据源类型不同”准备的，不是为“列名不同”准备的。

22. 怎么加一个新 adapter

如果你的数据根本不在 Hugging Face datasets 里，比如是本地 JSONL、数据库、对象存储清单，这时才考虑新 adapter。

判断标准

列名不同 → 新 spec

数据源类型不同 → 新 adapter

新 adapter 至少要做两件事：

  • 实现 __iter__()，持续 yield CanonicalSample
  • 实现 __len__()
class MyJsonlAdapter(DatasetAdapter):
    def __iter__(self):
        for row in load_jsonl(...):
            yield CanonicalSample(...)

    def __len__(self):
        return ...

但要注意，当前 Trainer 的正式路径仍然只接受 hf_datasets，所以如果你真的新增 adapter，还需要把：

  • training_config._validate()
  • trainer._build_dataset()
  • 可能还有 debug / smoke 路径里的 adapter 构造

一起扩进去。

这也是当前 data 包最值得继续演进的地方

架构已经为多 adapter 留了口子，但产品层面还没有把这条能力完全打通。这个差距在教学里说清楚，读者会更信任这份文档。
